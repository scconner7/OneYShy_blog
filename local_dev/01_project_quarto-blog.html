Project 01 — “Tidyverse to Python, Honestly” (Quarto Blog)

Priority: #1. This is the foundation. Every other project on the list gets writeups here, so the blog compounds value across the entire portfolio. Ship the scaffolding fast, then let it accumulate.

Mission

A polyglot technical blog aimed at R developers seriously evaluating or in the middle of moving to Python. Your differentiator is genuine fluency in both worlds — most “R vs Python” content is written by people who only know one side well. You know both, and you have a stack opinion (polars, plotnine, Shiny for Python, Quarto, uv) that’s coherent and defensible.

What “done with v1” looks like

  • Live at a custom-ish URL (e.g. seanconner.dev or <github-username>.github.io)
  • Quarto-native, deployed via GitHub Actions
  • Clean, opinionated theme (not the default)
  • 5 published posts
  • Working RSS, Giscus comments, simple analytics
  • New posts go live by pushing to main — no manual deploy steps

Why this matters

  • Compounding asset: every other portfolio project gets a blog post, which multiplies its discoverability.
  • Reputation signal: technical writing is the single most efficient way to demonstrate you can think, communicate, and ship. It’s worth more to a hiring manager than another GitHub repo.
  • Forcing function: writing about your projects forces you to actually finish them.
  • Aligns with your values: contributing to a professional community without self-promotion is exactly what a good technical blog does — you give away knowledge and let it speak for you.

Stack

Layer Tool
Site engine Quarto (website project)
Hosting GitHub Pages (free, fast, zero config)
Deploy GitHub Actions (quarto-dev/quarto-actions)
Comments Giscus (free, GitHub-discussions backed)
Analytics GoatCounter or Plausible (privacy-respecting)
Domain (optional) Cloudflare Registrar or Porkbun

Phase plan

Phase 1 — Scaffolding (1 evening, ~2 hours)

Goal: A barely-themed Quarto site live on GitHub Pages with one placeholder post.

Checkpoint: You can hit https://<user>.github.io/tidyverse-to-python-honestly/ and see your placeholder post.

Phase 2 — Make it look like yours (1 weekend, ~6 hours)

Goal: Site has a distinctive visual identity that isn’t the default Quarto theme.

    • Pick a serif for headings (e.g. Source Serif, Newsreader, EB Garamond)
    • Pick a sans for body (e.g. Inter, IBM Plex Sans)
    • Pick a mono for code (e.g. JetBrains Mono, Fira Code)
    • Limit yourself to 2 accent colors max

Checkpoint: Someone landing on the site can tell within 5 seconds what it’s about and who you are. The design looks intentional, not default.

Reference sites to study (don’t copy, but absorb): Albert Rapp, Isabella Velásquez, Andrew Heiss, Drew Breunig.

Phase 3 — The five cornerstone posts (3-4 weekends, ~20 hours)

Goal: Five high-quality posts that together form a coherent thesis: “here’s the modern Python data stack from an R-developer’s perspective.”

These are pre-scoped so you don’t have to invent topics. Tackle them in roughly this order:

  1. “Why I’m moving from pandas to polars (after 8 years of dplyr)”
    • Hook: dplyr’s mental model maps to polars almost 1:1, and to pandas not at all
    • Show 5-6 dplyr expressions, then polars equivalents, then pandas equivalents
    • Be honest about pandas’ strengths (ecosystem, ML library interop)
    • Effort: 1 weekend
  2. “plotnine: ggplot2 for Python, and it’s actually good”
    • Walk through porting a real ggplot2 chart to plotnine
    • Note the gaps (animation, some extensions) honestly
    • Bonus: side-by-side with seaborn and matplotlib equivalents
    • Effort: 1 weekend
  3. “Shiny for Python vs Shiny for R: a working developer’s comparison”
    • Build the same small app in both, link to repos
    • Cover reactive model differences, deployment story, ecosystem
    • Effort: 1 weekend (heavier — you’ll need to actually build both apps)
  4. “A modern Python project setup that feels like RStudio”
    • Basically a narrative version of your Positron + uv cheatsheet
    • Link to the cheatsheet as a companion reference
    • Effort: 1 evening (you already wrote the source material)
  5. “Quarto is the bridge: how I write code in two languages without switching tools”
    • Show a polyglot Quarto doc with R and Python in the same file passing data via Arrow
    • Lean on your actual experience
    • Effort: 1 evening

Checkpoint: 5 posts live, each with code, each at least 1,500 words. The blog now reads as a coherent body of work, not a collection of one-offs.

Phase 4 — Production polish (1 evening, ~3 hours)

Goal: All the small things that make it feel like a real, maintained blog.

Checkpoint: The blog is fully operational. Pushing to main deploys. Comments work. RSS works. Analytics are flowing.

Phase 5 — Cadence (ongoing)

The hardest phase, because no one’s grading it.

  • Commit to one post per month minimum. Set a calendar reminder on the 1st.
  • Keep a _drafts/ folder with seed ideas — never start from a blank page.
  • Tie posts to your other portfolio projects — every project below should produce 1-2 posts.
  • Allow yourself short posts. A 600-word post about a single sharp idea is more valuable than a 4,000-word post you never finish.

A reasonable trailing target: 12 posts in your first year, 20 in your second. That puts you in the top decile of technical bloggers by output alone.

Boss fights (anticipate these)

  • Theme rabbit hole. You will want to make the site perfect before writing. Don’t. The cornerstone posts matter 10× more than the typography. Set a 6-hour cap on Phase 2 and move on even if you’re not 100% satisfied.
  • Perfection in posts. First drafts of all 5 cornerstone posts should ship within 4 weekends. Polish in v2. Done > perfect.
  • Engine config friction. Polyglot Quarto (R + Python in the same doc) requires both knitr and jupyter available. Document the setup in your README so you don’t lose hours debugging it later.

Done criteria (the only ones that matter)

  1. Site is live on a domain you control.
  2. Five cornerstone posts published.
  3. Deploy is automated.
  4. You have a system (calendar reminder + drafts folder) for ongoing posts.

When all four are true, this project is shipped. Everything beyond that is maintenance and growth, which happens alongside the other projects.
